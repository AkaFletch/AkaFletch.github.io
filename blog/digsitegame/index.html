Digsite

Following LD47 I was pretty annoyed at my complete inability to use OpenGL since it caused a lot of wasted hours and on the time limit I didn’t have time to slow down and learn how to use it correctly. So following this I came up with a general idea for a game I want to make in 3D. It feels like a perfect practice ground for OpenGL and a perfect way to struggle my way through 3d game development.

The idea for Digsite is an adventure game about trying to be an archaeologist. I plan to make it follow a chapter format where each chapter is one individual “case” with their own little story line. Digsite is also just a project name, I’m not releasing it under that name, I can’t stop reading it as dogshit…

I want to keep making regular blog posts about the progress of making the game, but let’s see how that goes because I’m not promising it… Saying that I want to make them monthly.

Developing the game

I want to make the game myself with a custom engine. I don’t really care for unity or other consumer engines. I’ve never written an engine anywhere near the capability of consumer engines, but nor have I written a game which requires one. I want to make the engine myself mostly because I want to, but technically speaking I don’t think I need the overhead of any consumer engines.

I’m writing the game in C and I have a few reasons for that:

  • I like C.
  • I want to code more in C.
  • C can be compiled to all architectures I know.
  • I don’t see any benefits from C++ (Operator overriding is nice but it doesn’t sell me)

As I’ve said I’m learning OpenGL for this, but I’m hoping to not be limited to OpenGL. The plan is to make sure the graphics rendering is easy to port to another backend like Vulkan or Metal (Like I’m buying a Mac.) My friend James once again is subject to my irritating questions on all things C and OpenGL but I can’t imagine achieving anything I want without his help.

For now I’m opening windows with SDL2 and using it to handle my controller input. Again I have hopes to abstract this so that I don’t fall into a dependence upon it.

Platforms

Obviously I want to release the game on all PC platforms, I’ve also got a homebrew switch which isn’t doing too much, so I’m planning on making the game compile to it so that I know I’m pretty platform independent. If I can make the game compile to a homebrew switch I’m pretty happy the porting process to windows/mac or TempleOS should be easy.

Designing the game

For design I have a few ideas and I’ve been talking to my friend Jess who actually has a masters and experience in Cultural Heritage Management. She’ll be making sure I don’t depict the field in a totally inaccurate manner and actually have some cool fancy words that we can all learn along the way. They’ll probably be Greek words though and a pain for me to spell. Saying all this, I will take some liberties for the sake of gameplay and fun, I’m wondering if there is a way I could point this out to the player.

I’m aiming to not create a full design doc, I feel like that will require too much planning and restrict creativity while I program the game play. Maybe that’s a way of saying “I’m using an agile methodology to game design” but really this is how I write everything. I’m generally just going to aim for what feature I need next and keep improving the product until it’s something I’m happy with.
